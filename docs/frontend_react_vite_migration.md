# React + Vite Migration

Документ описывает текущее состояние новой операторской панели и правила ее дальнейшего развития.

## Цель миграции

Старая операторская панель собиралась постепенно и стала неудобной для дальнейшего развития.  
Новый frontend-контур переносится на `React + Vite + Ant Design`, чтобы:

- упростить поддержку интерфейса;
- разделить логику, представление и API-слой;
- легче внедрять новый дизайн;
- упростить ролевой доступ и работу со сложными таблицами, формами и drawer-сценариями.

## Базовый ориентир

В качестве визуального и архитектурного ориентира используется шаблон:
- `larry-xue/react-admin-dashboard`

Он используется как база, но не как готовый продукт без адаптации.

## Актуальные маршруты

- `/operator-next` — текущая рабочая React-операторка
- `/operator-template` — просмотр оригинального шаблона без правок
- `/operator` — старая HTML-версия, оставлена как переходный контур

## Ролевая модель

### `field_worker`
- видит только свой рабочий контур
- работает только со своими назначениями

### `operator`
- видит `Контроль`, `Журнал`, `Склады`, `Конфликты`, `Назначения`, `Группы`, `Экспорт`
- не видит вкладку `Сотрудники`

### `admin`
- видит все вкладки
- управляет сотрудниками и ролями

## Структура frontend-модуля

- `frontend/src/App.jsx` — основной orchestration-контейнер
- `frontend/src/lib/api.js` — API-слой
- `frontend/src/operator/constants.js` — интерфейсные константы
- `frontend/src/operator/utils.js` — чистые утилиты
- `frontend/src/operator/components.jsx` — переиспользуемые UI-компоненты
- `frontend/src/operator/tabs/` — изолированные вкладки
- `frontend/src/styles.css` — глобальные стили панели

## Правила дальнейшего развития

1. Не складывать новую бизнес-логику обратно в один большой `App.jsx`, если ее можно вынести в:
   - `constants`
   - `utils`
   - `components`
   - отдельные tab-модули
   - hooks
2. Все новые утилиты должны быть тестируемыми отдельно от React-компонентов.
3. Все новые экраны должны описываться не только кодом, но и коротко в документации.
4. После каждого заметного UI-изменения нужно проверять:
   - desktop
   - mobile
   - `light/dark`
   - роли

## Ближайший желательный шаг

- дальше выносить сценарную логику из `App.jsx` в hooks
- уменьшать связность shell и бизнес-логики
- расширять тесты сводок, role-based access и форм
- продолжать оптимизацию тяжелого `ui-vendor` chunk
