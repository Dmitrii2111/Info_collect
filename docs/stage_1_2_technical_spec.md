# Техническая спецификация: Этапы 1–2

## Контекст

После уточнения бизнес-правил система должна перейти от упрощенного складского контура к модели:

- приемка по строкам поступления и количеству;
- учет по позиции и количеству до размещения;
- создание экземпляров только при перемещении в помещения;
- автоматическая обработка недостачи, излишков и внеплановых позиций.

Новая ролевая схема:

- `operator` — полевой оператор;
- `dispetcher` — диспетчер;
- `admin` — супервайзер.

## Этап 1. Приемка на объект

### Цель

Реализовать полноценную приемку оборудования на объекте по строкам поступления без поэкземплярного учета.

### Бизнес-результат

Сотрудник или диспетчер может:

- создать поступление;
- занести строки поступления;
- отметить фактическое количество;
- зафиксировать состояние;
- приложить фото;
- сохранить комментарий;
- завершить приемку как подтвержденную.

### Что меняется в данных

#### WarehouseReceipt

Новые или уточненные поля:

- `receipt_no`
- `building_id`
- `target_storage_zone_id`
- `status_code`
- `created_by`
- `created_at`
- `confirmed_at`
- `comment_text`

#### WarehouseReceiptItem

Новые или уточненные поля:

- `position_code` — шифр/позиция из плана или из документа;
- `planned_position_id` — nullable;
- `equipment_name`;
- `model_mark`;
- `declared_quantity`;
- `actual_quantity`;
- `condition_status`;
- `completeness_status`;
- `status_code`;
- `placement_status`;
- `photo_refs_json`;
- `comment_text`.

#### Новые enum-ы

- `ConditionStatus`
  - `good`
  - `damaged`
  - `requires_inspection`
  - `incomplete`
  - `other`
- `ReceiptItemStatus`
  - `draft`
  - `confirmed`
  - `shortage`
  - `surplus`
  - `unplanned`
- `PlacementStatus`
  - `awaiting_placement`
  - `placed_to_stock`
  - `placed_to_room`
  - `partially_placed`

### API Этапа 1

#### `POST /api/stock/receipts`

Создание поступления с набором строк.

Тело:

- номер документа / внешняя ссылка;
- целевой склад или режим `awaiting_placement`;
- комментарий;
- строки:
  - `position_code`
  - `planned_position_id` nullable
  - `equipment_name`
  - `declared_quantity`
  - `actual_quantity`
  - `condition_status`
  - `completeness_status`
  - `photo_refs`
  - `comment_text`

#### `GET /api/stock/receipts/{receipt_id}`

Получение детальной карточки поступления.

#### `POST /api/stock/receipts/{receipt_id}/confirm`

Подтверждение приемки.

На этом этапе:

- валидируется фактическое количество;
- валидируется состояние;
- создаются доменные события;
- обновляется статус строк;
- формируются заготовки для конфликтов и излишков.

### Правила Этапа 1

1. Экземпляры оборудования не создаются.
2. Если склад явно не выбран, приемка попадает в виртуальную зону `awaiting_placement`.
3. Фото обязательны по каждой строке.
4. Состояние обязательно по каждой строке.
5. Приемка завершается после подтверждения количества и состояния.

## Этап 2. Недостача, излишки и внеплановые позиции

### Цель

Автоматизировать обработку расхождений приемки и научить систему учитывать излишки и внеплановое оборудование.

### Бизнес-результат

При подтверждении приемки система сама:

- фиксирует недостачу;
- создает конфликт;
- создает задание на допоставку;
- отправляет излишки в виртуальный склад `Излишки`;
- отправляет внеплановые позиции в виртуальный склад `Излишки`.

### Что меняется в данных

#### StorageZone

Добавляется тип зоны:

- `physical`
- `surplus`
- `awaiting_placement`

Виртуальные зоны:

- `Излишки`
- `Ожидает размещения`
- `Карантин`

создаются автоматически при необходимости.

#### StockBalance

Нужно поддержать хранение:

- либо по `planned_position_id`,
- либо по `warehouse_receipt_item_id`,
- либо по `planned_item_id` в более поздних этапах.

Это нужно, чтобы:

- in-plan остатки учитывались по позиции;
- внеплановые строки и излишки могли существовать до размещения без создания экземпляров.

#### Conflict

Добавляются или используются типы:

- `receipt_shortage`
- `receipt_surplus`
- `unplanned_receipt`

#### Follow-up задача

Новая сущность:

- `receipt_follow_up_tasks`

Поля:

- `id`
- `warehouse_receipt_item_id`
- `task_type`
- `required_quantity`
- `status_code`
- `created_at`
- `resolved_at`
- `created_by`
- `resolved_by`
- `comment_text`

Задача нужна для сценария допоставки.

### API Этапа 2

#### `GET /api/stock/receipts/{receipt_id}/issues`

Список проблем по поступлению:

- недостачи;
- излишки;
- внеплановые позиции;
- follow-up задачи.

#### `POST /api/stock/receipts/{receipt_id}/issues/{issue_id}/resolve`

Разрешение проблемы:

- подтвердить как излишек;
- перевести на склад;
- закрыть задачу после допоставки;
- отменить ошибочную фиксацию.

### Правила Этапа 2

1. Если `actual_quantity < declared_quantity`:
   - строка получает статус `shortage`;
   - создается конфликт;
   - создается задача на допоставку.
2. Если `actual_quantity > declared_quantity`:
   - заявленное количество учитывается как обычная приемка;
   - разница уходит в виртуальный склад `Излишки`;
   - строка получает статус `surplus`.
3. Если `planned_position_id` отсутствует:
   - вся строка считается внеплановой;
   - количество уходит в `Излишки`;
   - создается конфликт `unplanned_receipt`.
4. Диспетчер и супервайзер могут подтверждать и разрешать такие проблемы.

## Пограничные условия

1. Допускается один и тот же сотрудник с разными ролями, но в каждый момент времени активна только одна роль.
   Переключение рабочего контура между ролями должно в дальнейшем выполняться прямо в профиле пользователя.
2. Фото должны быть обязательными уже на этапе приемки.
3. Состояние не блокирует перемещение в помещение на следующих этапах, но проблемные позиции должны участвовать в конфликтах.
4. Полная история всех действий должна идти в `domain_events`.

## Что сознательно не входит в Этапы 1–2

Не входят:

- создание экземпляров;
- ввод серийных номеров;
- перемещения по помещениям;
- подтверждение перемещений;
- повторные проверки;
- инвентаризация складов как отдельное назначение.

Это идет следующими этапами после завершения приемки и расхождений.
