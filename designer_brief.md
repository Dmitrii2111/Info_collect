# Задание для дизайнера

## 1. Цель документа

Этот документ нужен для проектирования нового интерфейса системы учета и проверки оборудования в здании.
Дизайнеру не требуется придумывать бизнес-логику, права доступа, состав данных или сценарии работы с нуля.
Все это уже определено в текущем проекте и должно быть корректно отражено в дизайне.

Главная задача дизайнера:
- спроектировать понятную и визуально цельную систему экранов;
- сохранить существующую логику ролей и маршрутов;
- показать взаимосвязь экранов, данных и сценариев;
- предусмотреть ключевые рабочие, пограничные и ошибочные состояния;
- спроектировать интерфейс так, чтобы им было удобно пользоваться и на компьютере, и на мобильных устройствах.

Визуальный стиль, палитра, акценты, форма карточек, кнопок, иконок и общая атмосфера интерфейса выбираются дизайнером самостоятельно.
Но структура, состав экранов, логика переходов, роли, сценарии, ошибки и состояние данных должны соответствовать этому заданию.

## 2. Контекст продукта

Система используется для поэкземплярного учета оборудования в здании.
Есть исходная Excel-ведомость, по которой у каждого физического экземпляра оборудования есть плановое место установки.
Полевые сотрудники обходят помещения, проверяют наличие оборудования, заполняют данные по экземпляру и фиксируют отсутствие.
Оператор или диспетчер контролирует процесс с компьютера.
Система должна работать при плохой или отсутствующей связи внутри здания.

Продукт состоит из двух основных контуров:
- мобильный полевой интерфейс в формате PWA;
- операторская веб-панель.

## 3. Используемый стек

Дизайнер должен понимать, на какой технической базе будет реализовываться интерфейс.
Это влияет на глубину состояний, типы компонентов и ограничения по поведению.

### Backend
- `FastAPI`
- `PostgreSQL`
- `SQLAlchemy`
- `Alembic`
- `openpyxl`
- `Uvicorn`

### Frontend
- мобильный контур: `PWA`
- локальное офлайн-хранилище: `IndexedDB`
- новая операторская панель: `React + Vite + Ant Design`
- базовый визуальный и архитектурный ориентир: `larry-xue/react-admin-dashboard`

### Дополнительные особенности реализации
- обязательная авторизация;
- ролевой доступ;
- работа с локальным сервером;
- офлайн-режим для поля;
- синхронизация накопленных действий;
- импорт и экспорт Excel;
- история действий и журнал событий;
- повторные проверки;
- группы сотрудников;
- динамические складские зоны.

## 4. Бизнес-смысл системы

Система решает 4 связанные задачи:

1. Проверка наличия оборудования в помещениях.
2. Фиксация атрибутов каждого физического экземпляра.
3. Работа со складами, поступлениями и перемещениями.
4. Контроль работы сотрудников и подготовка отчетности.

Ключевая единица учета:
- не строка Excel;
- не тип оборудования;
- а физический экземпляр оборудования.

Если в исходной ведомости указано количество `3`, в системе это должны быть 3 отдельных экземпляра.

## 5. Роли и права доступа

### 5.1. Оператор

Это полевой сотрудник.

Доступ:
- только к своему кабинету;
- только к своим назначениям;
- только к мобильному полевому интерфейсу;
- к подтверждению приемки в рамках полевых операций;
- к просмотру своих действий и статуса синхронизации.

Нет доступа:
- к полной операторской панели;
- к управлению сотрудниками;
- к управлению ролями;
- к глобальному контролю;
- к настройке складов.

### 5.2. Диспетчер

Это офисный пользователь операторской панели.

Доступ:
- ко вкладке `Контроль`;
- ко вкладке `Назначения`;
- ко вкладке `Группы`;
- ко вкладке `Экспорт`;
- к управлению назначениями;
- к просмотру прогресса сотрудников;
- к работе с конфликтами;
- к работе со складами и отчетами;
- к просмотру данных по помещениям и экземплярам.

Нет доступа:
- к вкладке `Сотрудники`;
- к управлению пользователями и ролями.

### 5.3. Супервайзер

Это администратор системы.

Доступ:
- ко всем вкладкам операторской панели;
- к созданию, редактированию, отключению и восстановлению сотрудников;
- к назначению ролей;
- ко всем операциям диспетчера;
- к полному контролю по системе.

## 8. Подробное описание экранов

## 8.1. Экран авторизации

### Назначение
Стартовая точка системы.
Определяет, какой интерфейс увидит пользователь после входа.

### Обязательные элементы
- логотип/название системы;
- поле логина;
- поле пароля;
- показать/скрыть пароль;
- кнопка входа;
- текст ошибки авторизации;
- состояния загрузки.

### Ошибки и состояния
- пустой логин;
- пустой пароль;
- неверный логин или пароль;
- сервер недоступен;
- пользователь отключен;
- успешный вход.

### Переходы
- оператор -> мобильный полевой экран;
- диспетчер -> операторская панель без вкладки сотрудников;
- супервайзер -> операторская панель со всеми вкладками.

## 8.2. Личный кабинет / профиль

### Назначение
Показывает, кто вошел в систему, и дает возможность отредактировать собственные данные.

### Что должно быть
- аватар;
- ФИО;
- роль;
- телефон;
- email;
- редактирование профиля;
- замена фото;
- смена пароля;
- при необходимости — тема интерфейса.

### Важно
Профиль должен быть визуально и логически единым и доступным из любой роли.

## 8.3. Мобильный экран `Мои назначения`

### Назначение
Главный рабочий экран полевого сотрудника.

### Что должно быть
- список назначенных помещений;
- каскадные фильтры: этаж -> отделение -> помещение;
- цветовой статус помещения:
  - не начато;
  - в работе;
  - завершено;
  - повторная проверка;
- счетчики прогресса;
- индикатор онлайн/офлайн;
- кнопка синхронизации;
- блок локальной очереди.

### Важно
Сотрудник не должен видеть чужие назначения.
Если пользователь состоит в группе, он должен видеть общий прогресс по групповым отметкам.

### Состояния помещения
- не начато: нет ни одной отметки;
- в работе: есть отметки, но помещение не завершено;
- завершено: помещение закрыто;
- повторная проверка: помещение перепроверяется отдельно.

## 8.4. Карточка помещения

### Назначение
Показывает весь состав оборудования в выбранном помещении.

### Что должно быть
- код помещения;
- название помещения;
- этаж;
- отделение;
- статус помещения;
- список всех экземпляров;
- краткая информация, кто и когда уже что-то отмечал;
- возможность завершить помещение;
- маркер повторной проверки.

### По каждому экземпляру
- шифр/позиция;
- наименование;
- экземпляр 1, экземпляр 2 и т.д.;
- текущий статус;
- серийный номер;
- ПНР;
- коммуникации;
- кто последний менял;
- когда последний менял.

## 8.5. Карточка экземпляра / форма отметки

### Назначение
Внесение факта по конкретному физическому экземпляру.

### Поля
- наличие:
  - найден;
  - отсутствует;
  - не проверено;
- серийный номер;
- вариант `не предусмотрен`;
- ПНР:
  - не требуется;
  - не проведено;
  - проведено;
  - монтаж;
- коммуникации:
  - отсутствуют;
  - выполнены;
  - выполнены с ошибками;
  - не предусмотрены;
- комментарий.

### Состояния и ошибки
- обязательные поля не заполнены;
- дублирующийся серийный номер;
- попытка записать невозможный статус;
- запись сохранена локально без сети;
- запись синхронизирована;
- конфликт с другой записью.

## 8.6. Операторская панель: `Контроль`

### Назначение
Главный обзорный экран диспетчера и супервайзера.

### Что должно быть
- активная версия плана;
- сводка по помещениям;
- сводка по экземплярам;
- график прогресса по помещениям;
- график прогресса по экземплярам;
- гистограмма закрытия помещений по дням и сотрудникам;
- блок рабочих помещений;
- блок проблемных экземпляров.

### Блок `Рабочие помещения`
Должен показывать:
- список помещений;
- фильтры по этажу, отделению, помещению;
- фильтры по статусу;
- количество оборудования в помещении;
- статусы помещения;
- последние действия.

### Блок `Проблемные экземпляры`
Должен показывать:
- проблемные экземпляры оборудования;
- фильтры по этажу, отделению, помещению;
- фильтр типа проблемы:
  - все;
  - не проверенные;
  - отсутствующие;
  - конфликтные;
  - без серийного номера;
  - ПНР требует внимания.

### Важно
Этот экран не про редактирование.
Он про наблюдение, выявление проблем и переход в более детальные сценарии.

## 8.7. Операторская панель: `Сотрудники`

### Назначение
Полное управление пользователями системы.

### Доступ
Только супервайзер.

### Что должно быть
- список активных сотрудников;
- отдельный сворачиваемый блок неактивных сотрудников;
- краткая сводка по сотрудникам;
- карточка сотрудника;
- создание нового сотрудника;
- редактирование сотрудника;
- отключение сотрудника;
- восстановление сотрудника.

### Карточка сотрудника
Должна содержать:
- фото;
- ФИО;
- роль;
- телефон;
- email;
- кнопки редактирования;
- кнопка отключения или восстановления;
- при необходимости — краткую занятость.

### Создание сотрудника
Поля:
- логин;
- пароль;
- фамилия;
- имя;
- отчество;
- телефон;
- email;
- роль;
- фото.

### Валидации, которые дизайнер обязан предусмотреть в интерфейсе
- логин: только латиница и цифры;
- пароль: не менее 8 символов, верхний и нижний регистр, цифра, спецсимвол;
- фамилия/имя/отчество: кириллица, не менее 2 символов;
- телефон: формат РФ;
- email: стандартная валидация;
- все поля обязательны, кроме фото;
- красная подсветка ошибочного поля;
- текст ошибки под полем или рядом с ним;
- без валидных полей создание невозможно.

## 8.8. Операторская панель: `Назначения`

### Назначение
Распределение работы между сотрудниками.

### Что должно быть
Левая часть:
- список сотрудников;
- сводка по сотрудникам:
  - всего активных;
  - в работе;
  - свободны;
  - неактивны.

Правая часть:
- сводка по выбранному сотруднику:
  - назначено;
  - завершено;
  - в работе;
  - не начато;
- дерево назначений:
  - этаж;
  - отделение;
  - помещение.

### Логика дерева
- по умолчанию показываются только этажи;
- этаж раскрывается в отделения;
- отделение раскрывается в помещения;
- у каждого уровня есть чекбокс;
- если отмечен этаж, выбираются все помещения этажа;
- если отмечено отделение, выбираются все помещения отделения;
- частичный выбор должен иметь отдельное визуальное состояние;
- уже назначенные элементы должны подниматься выше, но не прыгать во время клика.

### Что должен понимать дизайнер
Этот экран не просто про выбор чекбоксов.
Он про:
- распределение фронта работ;
- контроль прогресса;
- предупреждение конфликтов;
- управление повторными проверками;
- групповую работу.

### Важные сценарии
1. Назначение помещений одному сотруднику.
2. Назначение пересекающихся помещений двум сотрудникам.
3. Предупреждение о пересечении.
4. Предложение:
   - отменить;
   - объединить в группу.
5. Назначение помещения, которое уже завершено.
6. Предупреждение, что доступна только повторная проверка.

### Состояния, которые дизайнер должен отразить
- помещение не назначено;
- помещение назначено;
- помещение в работе;
- помещение завершено;
- помещение требует повторной проверки;
- у сотрудника нет назначений;
- у сотрудника все назначения закрыты;
- у сотрудника есть назначения, но он еще не начал;
- назначение конфликтует с назначением другого сотрудника.

## 8.9. Операторская панель: `Группы`

### Назначение
Работа с ситуациями, когда несколько сотрудников проверяют общий набор помещений.

### Что должно быть
- список групп;
- краткая сводка по выбранной группе;
- состав группы;
- общий объем назначений группы;
- общий прогресс группы.

### Важная логика
Если сотрудники объединены в группу:
- отметки одного сотрудника должны отображаться у остальных;
- прогресс считается совместно;
- групповые назначения логически общие.

### Для дизайнера
Нужно показать:
- что такое группа;
- кто в нее входит;
- какие помещения для группы общие;
- как выглядит групповой прогресс;
- как отличить индивидуальные назначения от групповых.

## 8.10. Операторская панель: `Экспорт`

### Назначение
Табличное представление итоговых данных перед выгрузкой.

### Что должно быть
- фильтры по этажу;
- фильтры по отделению;
- фильтры по помещению;
- фильтр по оборудованию;
- фильтр по серийному номеру;
- фильтр по статусу;
- таблица;
- кнопка обновления;
- подготовка к Excel-экспорту.

### Основные колонки
- этаж;
- отделение;
- помещение;
- позиция;
- оборудование;
- экземпляр;
- наличие;
- серийный номер;
- ПНР;
- коммуникации;
- дата проверки;
- сотрудник.

### Важно
Это не просто отчет.
Это рабочее табличное представление данных, которое должно быть удобным для фильтрации, сверки и последующего экспорта.

## 8.11. Складские страницы и складские сценарии

### Назначение
Работа со складскими зонами, поступлениями и перемещениями.

### Что должен учитывать дизайнер
Сейчас складской функционал менее проработан визуально, но должен быть заложен в архитектуру.

Нужно предусмотреть будущие страницы:
- список складских зон;
- карточка складской зоны;
- поступления;
- подтверждения приемки;
- перемещения между складами;
- выдача со склада;
- история складских действий.

### Важные особенности
- складов может быть несколько;
- склады могут меняться со временем;
- один склад может быть перенесен, разделен или переименован;
- подтверждение приемки выполняется отдельно;
- полевые сотрудники тоже участвуют в процессе подтверждения.

## 8.12. Конфликты и события

### Назначение
Отдельный смысловой блок, который должен быть виден в системе как сущность.

Нужно предусмотреть интерфейсные решения для:
- конфликта данных по одному экземпляру;
- конфликта назначений;
- повторной проверки;
- просмотра журнала действий;
- просмотра истории изменения полей.

## 9. Взаимосвязь экранов

### Базовые переходы

1. Вход -> определение роли -> нужный интерфейс.
2. Контроль -> помещение -> состав помещения -> экземпляр.
3. Назначения -> сотрудник -> дерево этажей -> помещение.
4. Назначения -> пересечение -> диалог объединения в группу.
5. Сотрудники -> карточка сотрудника -> редактирование.
6. Экспорт -> фильтрация -> итоговая таблица.
7. Мои назначения -> помещение -> экземпляр -> сохранение -> синхронизация.

### Важно для дизайнера
Нужно не просто рисовать отдельные страницы.
Нужно проектировать систему переходов между ними.
Пользователь должен понимать:
- где он находится;
- почему он сюда попал;
- что он может сделать дальше;
- как вернуться на предыдущий уровень.

## 10. Основные сценарии использования

## 10.1. Оператор (полевой)
- входит в систему;
- видит свои назначения;
- открывает помещение;
- видит состав оборудования;
- вносит отметку;
- завершает помещение;
- при отсутствии связи сохраняет все локально;
- при появлении связи синхронизирует.

## 10.2. Диспетчер
- входит в операторскую панель;
- открывает `Контроль`;
- видит общий прогресс;
- заходит в `Назначения`;
- распределяет помещения;
- видит пересечения и предупреждения;
- смотрит `Группы`;
- переходит в `Экспорт` для сверки и выгрузки.

## 10.3. Супервайзер
- входит в операторскую панель;
- управляет сотрудниками;
- назначает роли;
- контролирует процесс как диспетчер;
- при необходимости редактирует сотрудников и доступы.

## 11. Ошибки и пограничные состояния, которые дизайнер обязан предусмотреть

### Общие
- пустые данные;
- ошибка загрузки;
- сервер недоступен;
- нет прав доступа;
- нет результатов по фильтру;
- длительная загрузка;
- сохранение прошло успешно;
- сохранение не прошло;
- конфликт данных;
- повторная проверка.

### Для формы сотрудника
- неверный логин;
- слабый пароль;
- неверное ФИО;
- неверный телефон;
- неверный email;
- фото не загружено;
- фото загружено;
- пользователь успешно создан;
- пользователь не создан.

### Для назначений
- сотрудник не выбран;
- нет доступных помещений;
- есть пересечение с другим сотрудником;
- назначаются уже закрытые помещения;
- частично выбран этаж;
- частично выбрано отделение.

### Для мобильного поля
- нет сети;
- есть локальная очередь;
- синхронизация успешна;
- синхронизация не удалась;
- помещение уже кем-то отмечалось;
- помещение повторно назначено.

## 12. Что важно в UX

- минимум лишних кликов в полевом сценарии;
- максимально понятная иерархия;
- явная навигация между уровнями;
- единая логика фильтров;
- единая логика карточек;
- ясные пустые и ошибочные состояния;
- без прыжков интерфейса;
- понятный ролевой доступ;
- визуально различимые:
  - рабочие состояния;
  - предупреждения;
  - ошибки;
  - завершенные действия;
  - повторные проверки;
  - конфликты.

## 13. Что дизайнер НЕ должен менять самостоятельно

Без согласования нельзя менять:
- ролевую модель;
- набор прав по ролям;
- базовую сущность "экземпляр оборудования";
- каскадную структуру этаж -> отделение -> помещение;
- обязательные поля формы сотрудника;
- обязательные поля отметки по экземпляру;
- офлайн-логику;
- механику пересечений назначений;
- механику групп;
- механику повторной проверки.

## 14. Что дизайнер должен выдать на выходе

Нужен следующий результат:
- карта экранов;
- user flow по ролям;
- wireframes;
- финальные макеты основных экранов;
- состояния экранов;
- ошибки и empty states;
- адаптивные версии;
- компоненты и UI-kit на уровне проекта;
- описание логики поведения для разработчика.

## 16. Финальное ожидание от дизайна

Дизайн должен не просто быть красивым.
Он должен стать рабочим интерфейсом для системы, где:
- есть несколько ролей;
- есть офлайн-сценарии;
- есть длинные деревья данных;
- есть контроль прогресса;
- есть конфликты;
- есть история;
- есть импорт и экспорт;
- есть мобильное поле и операторский контур.

Дизайн должен объяснять пользователю систему, а не усложнять ее.
