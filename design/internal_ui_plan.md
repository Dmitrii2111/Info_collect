# Internal UI Plan

## Цель

Этот документ фиксирует, как мы дальше развиваем интерфейс без участия дизайнера.

Он заменяет цикл `docx -> feedback -> новая версия` и становится нашей внутренней опорой для:

- визуальной доработки экранов
- уточнения UX
- реализации новых модулей
- синхронизации с ТЗ и текущим кодом

## Источники, которые берем в работу

### Берем как основу

- утвержденное ТЗ
- текущую реализацию проекта
- сильные решения из `Design_V3`
- очищенную терминологию и часть мобильного контура из `Design_V4`

### Визуальная основа

По визуалу, структуре экранов и цветовым решениям за базу берем:

- `Design_V3`
- `Design_V4`

Именно эти две версии считаются опорными для:

- композиции экранов
- общей визуальной иерархии
- цветовой логики состояний
- мобильного темного контура
- десктопного рабочего стиля панели

### Не используем как источник решений

- `Design_V5`

Причина:
- документ содержит смысловые регрессии
- снова смешивает разные сущности экранов
- не может считаться надежной основой для продуктовых решений

## Общие UI-принципы

### 1. Интерфейс должен быть предметным, а не шаблонным

Мы не проектируем “абстрактную админку”, а рабочую систему учета оборудования.

Следовательно:
- названия экранов и действий должны быть из предметной области
- на каждом экране пользователь должен видеть только нужные для этого сценария данные
- запрещено смешивать сущности между модулями

### 2. Один экран — одна основная задача

Нельзя смешивать:
- сотрудников и оборудование
- склады и назначения
- контроль и редактирование учеток

### 3. Приоритет на читаемость и рабочий поток

Визуал должен помогать быстро:
- понять статус
- найти проблему
- выполнить действие
- увидеть последствия действия

### 4. Мобильный и десктопный контуры проектируются отдельно

PWA для поля не должен повторять логику desktop-таблиц.

## Роли и доступ

## Авторизация

Авторизация обязательна для всех ролей без исключения.

Через экран входа в систему проходят:

- Супервайзер
- Диспетчер
- Полевой сотрудник

После авторизации пользователь попадает в интерфейс, соответствующий своей роли и правам доступа.

Без авторизации недоступны:

- операторская панель
- мобильный полевой контур
- профиль
- рабочие данные по помещениям, экземплярам, складам и событиям

### Супервайзер

Видит:
- Контроль
- Сотрудники
- Назначения
- Группы
- Экспорт
- Склады
- Конфликты
- Журнал событий
- Профиль

### Диспетчер

Видит:
- Контроль
- Назначения
- Группы
- Экспорт
- Склады
- Конфликты
- Журнал событий
- Профиль

Не видит:
- Сотрудники

### Полевой сотрудник

Видит только PWA-контур:
- Авторизация
- Мои назначения
- Помещение
- Экземпляр
- Профиль
- Очередь синхронизации

## План по экранам

## 1. Контроль

### Назначение

Главный монитор состояния объекта.

### Что должно быть на экране

- сводка по помещениям
- сводка по экземплярам
- график прогресса по дням
- проблемные помещения
- проблемные экземпляры
- фильтры по этажу / отделению / помещению / статусу

### Что берем за основу

- из текущей реализации: общую структуру вкладки и сводки
- из `V3/V4`: идею предметного, а не шаблонного мониторинга

### Что улучшать сами

- сильнее развести уровни `помещение / экземпляр / конфликт`
- уменьшить визуальный шум
- доработать плотность данных без перегруза
- сделать более сильный акцент на проблемных состояниях

### Приоритет

Высокий

---

## 2. Сотрудники

### Назначение

Управление учетными записями пользователей системы.

### Что должно быть на экране

- список сотрудников
- краткая сводка
- создание сотрудника
- редактирование сотрудника
- фото
- логин
- пароль / смена пароля
- телефон
- email
- роль
- активен / неактивен
- восстановление сотрудника

### Что берем за основу

- из текущей реализации: форма, валидации, роли, фото, отключение/восстановление

### Что запрещено смешивать с этим экраном

- серийный номер
- ПНР
- коммуникации
- статусы оборудования
- история экземпляра

### Что улучшать сами

- сделать экран более цельным и менее формальным
- улучшить hierarchy списка сотрудников
- улучшить layout модалки редактирования
- сделать явнее разграничение `активные / неактивные`

### Приоритет

Очень высокий

---

## 3. Назначения

### Назначение

Распределение помещений между сотрудниками и контроль прогресса по назначениям.

### Что должно быть на экране

- список сотрудников
- статусы сотрудников
- дерево `этаж -> отделение -> помещение`
- чекбоксы всех уровней
- partial state / indeterminate
- статусы помещений:
  - не начато
  - в работе
  - завершено
  - повторная проверка
- предупреждение о пересечениях
- объединение в группу
- повторное назначение завершенных помещений

### Что берем за основу

- из текущей реализации: рабочую логику дерева и пересечений
- из `V3/V4`: структуру сценария назначения и конфликта

### Что улучшать сами

- сделать дерево визуально чище
- усилить читаемость статусов
- улучшить отображение назначенных помещений
- сделать сценарии пересечения и повторной проверки более наглядными

### Приоритет

Очень высокий

---

## 4. Группы

### Назначение

Совместная работа нескольких сотрудников по общему набору помещений.

### Что должно быть на экране

- список групп
- состав группы
- общие помещения группы
- общий прогресс
- просмотр участников

### Что берем за основу

- из текущей реализации: группировку и сводку
- из `V3`: сам факт наличия отдельного модуля

### Что улучшать сами

- сделать модуль визуально самостоятельным
- показать отличие группы от обычного назначения
- лучше отрисовать общий прогресс и область ответственности

### Приоритет

Средний

---

## 5. Экспорт

### Назначение

Рабочий просмотр плоской таблицы перед выгрузкой результатов.

### Что должно быть на экране

- фильтры
- таблица
- цветовая индикация строк
- дата проверки
- сотрудник
- статусы

### Что берем за основу

- из текущей реализации: таблицу и фильтры
- из `V3/V4`: идею продукта как “рабочей сверки перед Excel”, а не просто таблицы

### Что улучшать сами

- усилить читаемость таблицы
- улучшить sticky headers / плотность
- сделать визуально ближе к реальному “операторскому” экрану

### Приоритет

Средний

---

## 6. Склады

### Назначение

Управление складскими зонами и складскими операциями.

### Что должно быть на экране

- список складских зон
- остатки
- поступления
- подтверждение приемки
- перемещения между складами
- выдача в помещение
- история операций

### Что берем за основу

- только общую идею отдельного складского модуля

### Что проектируем сами

Почти весь модуль, потому что ни одна из дизайнерских версий не раскрыла его достаточно точно.

### Приоритет

Высокий, но после доработки `Сотрудников` и `Назначений`

### Что улучшать сами

Средний
