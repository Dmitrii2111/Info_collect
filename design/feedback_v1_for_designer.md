# Обратная связь по дизайн-проекту V1

## Статус

`V1 concept approved with major revision needed`

Общее направление можно считать удачным:
- выбран понятный enterprise-подход;
- десктопная часть визуально близка к операторской системе;
- `drawer` для карточки экземпляра выглядит уместно;
- идея `audit trail / timeline` подходит под нашу событийную модель;
- в мобильной части есть попытка показать офлайн-режим и синхронизацию.

При этом текущая версия пока воспринимается как хороший концепт, но не как полный дизайн реального продукта.
В макетах еще недостаточно отражены реальные роли, сценарии, продуктовые разделы и критичные состояния системы.

## Что понравилось

1. Десктопная операторская часть идет в правильную сторону: плотная таблица, фильтры, правая карточка деталей.
2. Использование `drawer` вместо тяжелой модалки для карточки экземпляра подходит нашему рабочему сценарию.
3. История изменений в виде `timeline` соответствует логике журнала событий.
4. Для мобильной части выбран более контрастный и прикладной подход, что может хорошо сработать в поле.
5. Цветовая логика статусов в целом читается:
   - найдено;
   - отсутствует;
   - конфликт;
   - внимание.

## Критичные замечания

### 1. Дизайн пока не покрывает реальную структуру продукта

В текущих макетах показаны в основном обобщенные разделы:
- `Dashboard`
- `Equipment`
- `Rooms`
- `Warehouse`
- `Conflicts`
- `Settings`

Для нашего продукта этого недостаточно.
В следующей итерации обязательно должны быть отрисованы реальные страницы системы:
- экран входа;
- личный кабинет / профиль;
- `Контроль`;
- `Сотрудники`;
- `Назначения`;
- `Группы`;
- `Экспорт`;
- `Мои назначения` для полевого сотрудника;
- карточка помещения;
- карточка экземпляра;
- складские страницы.

### 2. Не раскрыта ролевая модель

Это один из самых важных аспектов проекта.
Сейчас по макетам нельзя понять:
- чем отличается интерфейс `оператора`;
- чем отличается интерфейс `диспетчера`;
- чем отличается интерфейс `супервайзера`.

В следующей итерации нужно явно показать:
- какие разделы доступны каждой роли;
- какие действия доступны каждой роли;
- какие экраны скрыты для разных ролей.

### 3. Отсутствует полноценный экран `Назначения`

Для нас это один из ключевых разделов системы.
Он должен быть показан не общо, а как рабочий сценарий с полной логикой.

Обязательно нужно отрисовать:
- список сотрудников;
- сводку по сотрудникам;
- дерево `этаж -> отделение -> помещение`;
- чекбоксы на каждом уровне;
- частичное выделение;
- сворачивание/раскрытие этажей и отделений;
- прогресс по сотруднику:
  - назначено;
  - завершено;
  - в работе;
  - не начато;
- предупреждение о пересечении назначений;
- предложение объединения в группу;
- повторную проверку уже закрытых помещений.

### 4. Мобильный полевой сценарий показан слишком поверхностно

Сейчас в мобильной части виден только общий список и блок синхронизации.
Этого недостаточно.

Нужно обязательно показать:
- экран `Мои назначения`;
- карточку помещения;
- список экземпляров в помещении;
- форму отметки по экземпляру;
- статусы помещения:
  - не начато;
  - в работе;
  - завершено;
  - повторная проверка;
- индикацию офлайн-режима;
- локальную очередь;
- отображение того, кто и когда уже вносил отметки по помещению или экземпляру.

## Замечания высокого приоритета

### 5. Слишком много шаблонной или англоязычной терминологии

Сейчас в макетах присутствуют шаблонные названия и англоязычные элементы:
- `Dashboard`
- `Equipment`
- `Warehouse`
- `Settings`
- `Sign In`
- `Scan PWA QR Code`

Нам нужен предметный интерфейс на языке продукта, а не generic admin-template.

Нужно перейти к реальной терминологии системы:
- `Контроль`
- `Сотрудники`
- `Назначения`
- `Группы`
- `Экспорт`
- `ПНР`
- `Коммуникации`
- `Повторная проверка`
- `Складские зоны`
- `Проблемные экземпляры`
- `Рабочие помещения`

### 6. QR-вход сейчас выглядит лишним

В утвержденной логике продукта QR-сценарий входа не является обязательным.
Поэтому на текущем этапе он либо не нужен вовсе, либо должен быть вынесен как отдельная опциональная функция на будущее.

Пока он перегружает экран авторизации и добавляет механику, которой нет в приоритетной реализации.

### 7. Карточка экземпляра не покрывает весь обязательный набор данных

Сейчас направление верное, но нужно обязательно учесть полный набор полей:
- наличие;
- серийный номер;
- значение `не предусмотрен`;
- ПНР;
- коммуникации;
- комментарий;
- источник изменения;
- устройство / web / mobile;
- offline / online происхождение;
- конфликт;
- повторная проверка;
- история изменения полей.

### 8. Складской модуль пока слишком упрощен

Склад в нашем продукте — это не просто перечень зон и строк с оборудованием.
Нужно заложить сценарии:
- несколько складских зон;
- поступление;
- подтверждение приемки;
- перемещение между складами;
- выдача в помещение;
- история операций;
- изменение структуры складов со временем.

## Замечания среднего приоритета

### 9. Нужно убрать технический и генеративный шум

В текущих экранах и описании встречаются артефакты, которые нельзя оставлять в следующей итерации:
- `Design armocampncnin`
- `Alert type="warning"`
- `PHP Status`
- `Server status: Calred`
- случайные или битые подписи
- неочищенные шаблонные тексты

Все надписи должны быть:
- осмысленными;
- предметными;
- чистыми;
- без template-шума.

### 10. Не хватает ошибок и пограничных состояний

Для нас это обязательная часть дизайна.
В следующей итерации нужно показать:
- пустые состояния;
- нет прав доступа;
- нет данных по фильтру;
- ошибка загрузки;
- конфликт данных;
- пересечение назначений;
- повторная проверка;
- дублирующийся серийный номер;
- нет сети;
- данные сохранены локально;
- очередь ожидает синхронизации;
- импорт с ошибками.

### 11. Нужен акцент на двух уровнях учета: помещение и экземпляр

Сейчас в концепте больше внимания уделено таблице оборудования.
Но в нашем продукте важно четко разделять:
- прогресс по помещениям;
- прогресс по экземплярам.

Дизайн должен это объяснять визуально.

## Что обязательно показать в следующей итерации

### Десктоп

1. `Контроль`
2. `Сотрудники`
3. `Назначения`
4. `Группы`
5. `Экспорт`
6. карточка экземпляра
7. карточка помещения
8. складские страницы
9. профиль пользователя

### Мобильный контур

1. вход
2. `Мои назначения`
3. карточка помещения
4. карточка экземпляра
5. очередь синхронизации
6. офлайн-режим
7. повторная проверка

### Отдельно по ролям

Нужно явно показать интерфейсы для:
- оператора;
- диспетчера;
- супервайзера.

## Что важно сохранить

1. `Ant Design` как визуальный и компонентный ориентир для десктопа.
2. `Drawer` как основной способ показывать карточку экземпляра без потери контекста таблицы.
3. `Timeline / Audit trail` как ключевой блок истории изменений.
4. Высокую информационную плотность в операторском интерфейсе.
5. Контрастный прикладной характер мобильного интерфейса.

## Рекомендация по следующей итерации

Предлагается следующий формат `V2`:

1. Сначала карта экранов по ролям.
2. Затем полный набор ключевых страниц.
3. Затем набор ошибок и крайних состояний.
4. Затем сценарии:
   - назначение;
   - пересечение;
   - объединение в группу;
   - повторная проверка;
   - офлайн и синхронизация;
   - конфликт данных.

## Итоговая оценка

Текущий дизайн-проект можно считать:

`Концептуально удачным, но недостаточно полным для передачи в разработку без следующей итерации.`

Следующая версия должна быть не просто красивым развитием шаблона, а полноценным проектным дизайном именно под нашу систему учета оборудования.
